# NovelScript AI 第一版产品需求

版本：v0.1 MVP  
项目名称：NovelScript AI  
定位：可追溯、可校验、可编辑的 AI 小说剧本改编工作台

## 1. 产品目标

NovelScript AI 面向小说作者和内容创作者，帮助用户把 3-5 个章节的小说文本转换为结构化剧本初稿。

第一版重点不是“让 AI 直接写剧本”，而是建立一条稳定的长文本改编流程：

```text
输入文件 → Markdown → 章节确认 → 段落编号
                         ↓
        ┌────────────────┴────────────────┐
        ↓                                 ↓
章节摘要 / 原文证据索引             风格来源处理
        ↓                                 ↓
  Story Bible                      Style Profile
        └────────────────┬────────────────┘
                         ↓
                    Scene Plan
                         ↓
                  用户确认 Scene Plan
                         ↓
                    逐场生成剧本
                         ↓
                    校验与修复
                         ↓
         必选导出 YAML，可选导出其他格式
```

执行规则：章节摘要、原文证据索引和风格来源处理可以并行执行；风格来源处理包括上传参考剧本分析或内置风格生成。Story Bible 需要在章节摘要和证据索引完成后生成；Scene Plan 需要在 Story Bible、Style Profile 和相关证据完成后生成。

## 2. 核心原则

1. 原始材料完整保存，不做删除式裁剪。
2. 长文本通过摘要、结构化记忆和证据索引进入生成流程。
3. 生成过程必须可追踪，开发者能查看 Prompt、工具调用、工具返回和校验报告。
4. 剧本生成过程必须可追溯，内部数据需要绑定原文证据，但用户最终导出文件默认不包含来源证据编号。
5. 内部优先使用 JSON，最终必须输出符合 Schema 的 YAML。
6. 用户可以通过对话持续修改剧本，而不是只能一次性生成。

## 3. MVP 范围

### 3.1 必须支持

- 用户可以新建项目对话。
- 上传小说文本并转换为 Markdown。
- 支持 3-5 个章节的小说改编。
- 自动识别章节，并允许用户确认章节划分。
- 给章节正文建立稳定段落编号。
- 生成章节结构化摘要。
- 生成原文证据索引。
- 生成 Story Bible。
- 支持上传过往剧本作为风格参考，或选择系统内置风格类型。
- 生成 Style Profile。
- 生成 Scene Plan，并允许用户确认或编辑。
- 逐场生成结构化剧本 JSON。
- 使用 Validation Agent 校验内容质量。
- 使用程序化 Schema Validator 校验结构。
- 修复失败内容并限制重试次数。
- 必选导出 YAML。
- 可选导出 Markdown、DOC/DOCX、PDF、TXT、JSON。
- 保存 session、checkpoint、execution log 和 developer run logs。
- 支持断点恢复和版本回退。

### 3.2 暂不支持

- 整本百万字小说一次性改编。
- 扫描版 PDF OCR。
- 图片、漫画、视频输入。
- 多人协作编辑。
- 真正训练或微调用户专属模型。
- 专业影视分镜生成。
- 配音、视频、音频生成。
- 复杂出版级排版。

## 4. 用户角色

### 4.1 小说作者

希望将已有小说改编为剧本初稿，降低改编门槛。

### 4.2 内容创作者

希望快速生成短剧、网剧或影视剧本雏形，再继续人工修改。

## 5. 输入需求

### 5.1 支持格式

第一版支持：

- Markdown：`.md`
- 文本文档：`.txt`
- Word 文档：`.doc`、`.docx`
- PDF：`.pdf`

PDF 限制：仅保证普通文本型 PDF 的正文提取，不保证扫描版 PDF、图片型 PDF 或复杂排版 PDF 的准确解析。

### 5.2 输入转换规则

所有输入文件先进入 Input Adapter，统一转换为 Markdown。

```text
doc/docx/txt/pdf/md
→ Input Adapter
→ Markdown
→ Markdown Normalizer
→ normalized_chapters.json
```

转换时需要：

- 保留段落边界。
- 保留章节标题。
- 清理明显无意义空行。
- 不删除短对白段落。
- 不直接修改原始文件。
- 保存原始文件和转换后的 Markdown。

### 5.3 上传模式

支持两种小说上传模式：

1. 单文件多章。
2. 多文件单章。

多文件上传时需要自然排序，并允许用户手动调整顺序。

## 6. 章节与段落处理

### 6.1 章节识别

系统应识别常见章节标题：

- 第一章
- 第1章
- Chapter 1
- CHAPTER 1
- 一、标题
- 1. 标题
- 序章、楔子、番外

识别后进入章节确认页，用户可以：

- 确认章节。
- 修改章节标题。
- 调整章节顺序。
- 重新识别。

### 6.2 段落编号

确认章节后，系统为每个段落生成稳定编号。

示例：

```text
CH001_P001
CH001_P002
CH002_P001
```

段落编号用于：

- 原文证据索引。
- 场景规划。
- 剧本来源追踪。
- 校验报告。
- 断点恢复。

段落编号生成后不得随意重排。

```text
raw_files/

paragraph_index.json
chapter_summaries.json
evidence_index.json
story_bible.json
style_profile.json
scene_plan.json
script_scenes/*.json
final_script.json
final_script.yaml
traceability_index.json
schema_document.md
adaptation_report.json
execution_log.jsonl
developer_runs/
checkpoints/
```

这些产物既用于生成，也用于透明化展示、回放、恢复和答辩说明。

数据库是结构化业务数据的权威数据源。文件系统主要保存原始文件、导出文件和开发者调试快照。若数据库记录与调试快照不一致，以数据库记录为准。

## 8. 技术栈与数据存储

### 8.1 技术栈

第一版技术栈约束：

- 后端语言：Python。
- 运行环境：Docker。
- 主数据库：PostgreSQL。
- 结构化中间产物：优先使用 PostgreSQL JSONB 存储。
- 原始文件、导出文件和开发者调试快照：使用文件存储保存，数据库保存元数据、索引和追溯关系。

### 8.2 PostgreSQL 存储范围

PostgreSQL 至少需要保存：

- 项目。
- session。
- 对话 conversation。
- 剧本生成 script generation。
- 上传文件元数据。
- 章节。
- 段落编号。
- 原文证据。
- Story Bible。
- Style Profile。
- Scene Plan。
- 剧本版本。
- checkpoint。
- run 记录。
- execution log。
- 导出记录。

章节摘要、Story Bible、Style Profile、Scene Plan、剧本 JSON、校验报告、checkpoint snapshot 和 run 摘要可以使用 JSONB 字段保存。

PostgreSQL 是 session、checkpoint、中间产物、剧本版本、run 记录和追溯关系的权威数据源。文件系统中的 JSON、YAML 或 JSONL 文件只作为原始材料、导出结果或开发者调试快照使用。

### 8.3 原始文件追溯保存

用户上传的原始文件必须保存，不能只保存解析后的文本。

原始文件包括：

- 小说原文件。
- 用户上传的已有剧本。
- 风格参考剧本。
- 导出的 YAML、Markdown、DOC/DOCX、PDF、TXT、JSON 文件。

数据库需要保存每个文件的元数据：

- 文件 ID。
- 所属 project ID。
- 所属 session ID。
- 原始文件名。
- 文件类型。
- 文件大小。
- 文件 hash。
- 存储路径。
- 上传时间。
- 转换状态。
- 对应的 Markdown 文件 ID。
- 是否作为风格参考。

后续所有章节、段落、证据、剧本场景和导出结果都应能追溯到对应原始文件。

### 8.4 开发者 run 日志文件夹

Prompt、工具调用参数、工具返回结果和模型输出主要供开发者排查问题，不作为普通用户默认展示内容。

每次用户发起一次生成、重新生成、校验、修复或对话修改时，系统必须创建一个独立 `run_id`。

每个 `run_id` 对应一个独立开发者日志目录：

```text
developer_runs/{run_id}/
  run_meta.json
  prompt.md
  prompt_injection_package.json
  model_output.json
  tool_calls.jsonl
  tool_results.jsonl
  validation_report.json
```

数据库需要保存 `run_id` 与 project、script_generation、conversation、session、checkpoint 的关联关系。

开发者 run 日志需要避免保存 API key、数据库连接串、系统密钥和其他敏感凭据。

### 8.5 Docker 环境要求

Docker 环境至少包含：

- Python 后端服务。
- PostgreSQL 数据库服务。
- 文件存储挂载目录。

第一版可以使用本地 Docker Compose 管理后端和数据库。数据库数据目录和文件存储目录必须持久化挂载，避免容器重启后丢失 session、checkpoint 和原始文件。

## 9. 章节结构化摘要

章节摘要不是普通摘要，而是服务剧本改编的结构化信息。

每章摘要应包含：

- 一句话摘要。
- 详细摘要。
- 本章叙事功能。
- 本章改编价值。
- 主要人物。
- 人物关系变化。
- 重要地点。
- 关键剧情点。
- 戏剧冲突。
- 情绪节点。
- 对白候选。
- 视觉元素。
- 伏笔线索。
- 适合改编的场景。
- 改编建议。
- 可能遗漏风险。

如果章节过长，先按段落分块生成 block summary，再合并为 chapter summary。

## 10. 原文证据索引

系统必须建立 evidence_index，用于减少遗漏和幻觉。

每条证据应包含：

- 证据编号。
- 来源章节。
- 相关段落编号。
- 原文摘录。
- 证据类型。
- 证据说明。
- 关联人物。
- 关联地点。
- 关联剧情点。
- 重要性。
- 是否必须保留。

支持的证据类型：

- 关键事件。
- 对白。
- 人物目标。
- 人物关系。
- 心理描写。
- 视觉元素。
- 线索。
- 伏笔。
- 冲突。
- 地点描写。

关键剧情点如果没有证据支撑，应在校验报告中标记为风险。

## 11. Story Bible

Story Bible 是项目级故事记忆，用于保持人物、关系、地点、时间线和伏笔一致。

必须包含：

- 故事标题。
- 故事类型。
- 整体基调。
- 一句话故事。
- 核心主题。
- 主要人物表。
- 人物关系表。
- 重要地点表。
- 剧情时间线。
- 主线冲突。
- 伏笔与线索。

Story Bible 不直接写剧本，只作为生成约束和校验依据。

## 12. 风格参考与 Style Profile

### 12.1 风格来源

用户创建项目时需要选择一种风格来源：

1. 上传过往剧本作为风格参考。
2. 选择系统内置风格类型。

两种方式第一版互斥，不同时启用。

第一版最多支持上传 3 份参考剧本。

如果用户上传参考剧本，系统从参考剧本中提取 Style Profile。

如果用户选择系统内置风格类型，系统根据所选风格生成默认 Style Profile。

系统不会训练模型，而是在当前 session 中生成或提取 Style Profile。

### 12.2 支持风格类型

第一版支持以下风格类型：

- 现实主义。
- 悬疑/惊悚。
- 爱情/情感。
- 喜剧。
- 短剧/网剧。

### 12.3 Style Profile 内容

Style Profile 应包含：

- 剧本类型。
- 对白密度。
- 场景长度。
- 节奏快慢。
- 角色说话风格。
- 冲突设计方式。
- 旁白比例。
- 动作描写比例。
- 常见转场方式。
- 用户偏好。

生成或修改时，系统只注入相关风格规则和必要参考片段，不把全部参考剧本直接塞入 Prompt。

如果用户需要从上传参考剧本切换为内置风格，或从内置风格切换为上传参考剧本，系统应生成新的 Style Profile 版本，并保存 checkpoint。

## 13. Scene Plan

系统生成剧本前必须先生成 Scene Plan。

Scene Plan 用于决定小说内容如何拆成剧本场景。小说章节和剧本场景不是一一对应关系。

每个场景应包含：

- 场景编号。
- 场景标题。
- 场景顺序。
- 来源章节。
- 来源证据编号。
- 地点。
- 时间。
- 内外景。
- 出场人物。
- 场景功能。
- 核心冲突。
- 必须覆盖的剧情。
- 必须保留的对白。
- 必须保留的视觉元素。
- 必须保留的伏笔。
- 改编说明。

用户需要在生成剧本前确认 Scene Plan。

确认前，Scene Plan 属于可协商阶段。用户可以通过对话提出调整要求，例如“我不喜欢这个规划，我想要节奏更快”“减少感情戏，增加悬疑感”“把前三章压缩成更少的场景”。系统应根据用户反馈重新生成或局部调整 Scene Plan，并保存新的 Scene Plan 版本。

Scene Plan 确认后进入剧本生成阶段。此后用户仍可通过对话修改剧本，但默认只修改已确认 Scene Plan 下的剧本内容，不隐式改变 Scene Plan 的场景数量、顺序、来源证据和核心冲突。

如果用户在剧本生成后明确要求改变场景结构，例如“删掉第二场”“把这两场合并”“重新规划场景”，系统应提示这属于 Scene Plan 变更，需要回到 Scene Plan 阶段并重新确认后再继续生成。

## 14. Agent 架构

第一版采用主控 Agent 编排的多 Worker 工作流，并保留独立 Validation Agent：

```text
Orchestrator Agent
+ Specialized Workers
+ Validation Agent
+ Programmatic Validators
```

### 14.1 Orchestrator Agent

负责：

- 编排整体流程。
- 管理 session、checkpoint 和调用次数。
- 组织每次 LLM 调用的 Prompt 注入包。
- 调度 Specialized Workers。
- 根据用户对话修改剧本。
- 根据校验报告执行修复。

### 14.2 Specialized Workers

Specialized Workers 是受控任务执行单元，可以拥有独立 Prompt、工具和输出格式，但不自主循环、不自行规划、不越权调用其他 Worker。

第一版包含：

- Chapter Summary Worker：生成章节结构化摘要。
- Evidence Extraction Worker：生成原文证据索引。
- Style Profile Worker：分析风格参考剧本。
- Story Bible Worker：汇总人物、关系、地点、时间线和伏笔。
- Scene Plan Worker：生成或调整 Scene Plan。
- Script Generation Worker：逐场生成剧本 JSON。

### 14.3 Validation Agent

使用不同模型，只负责检查，不负责自由创作。

负责：

- 检查剧情覆盖率。
- 检查角色一致性。
- 检查场景是否符合 Scene Plan。
- 检查是否遗漏关键证据。
- 检查是否出现严重幻觉。
- 检查是否符合 Style Profile。
- 输出结构化校验报告。

Validation Agent 不直接改剧本。

### 14.4 Programmatic Validators

负责确定性校验：

- JSON Schema 校验。
- YAML Schema 校验。
- 必填字段校验。
- 编号唯一性校验。
- 引用 ID 是否存在。
- YAML 是否可解析。

## 15. 剧本生成

系统逐场生成剧本，不一次性生成完整剧本。

单场生成上下文应包含：

- 当前 Scene Plan。
- 相关章节摘要。
- 相关原文证据。
- Story Bible。
- Style Profile。
- Schema 约束。
- 用户当前修改要求。

内部输出为 JSON，最后转换为用户可查看的 YAML。

单场 JSON 至少包含：

- 场景编号。
- 场景标题。
- 来源章节。
- 来源证据。
- 场景信息。
- 出场人物。
- 场景目的。
- 核心冲突。
- 内容块列表。

内容块类型：

- 动作。
- 对白。
- 旁白。
- 转场。
- 注释。

对白必须包含说话人。

来源证据 ID 保留在内部 JSON、traceability_index 和页面追溯层中，用于校验、回放和弹窗查看原文，不默认进入用户最终导出文件。

## 16. 对话式修改

用户可以通过自然语言持续修改剧本。

对话式修改分为两个阶段：

1. Scene Plan 确认前：用户对话修改的是场景规划，系统可以重新生成或调整 Scene Plan。
2. Scene Plan 确认后：用户对话修改的是剧本内容，系统默认沿用已确认的 Scene Plan，不主动改变场景结构。

支持修改范围：

- 全剧。
- 某一章。
- 某一场。
- 某个角色。
- 某段对白。
- 某类风格。

每次修改需要：

- 解析用户意图。
- 确定修改范围。
- 判断修改发生在 Scene Plan 确认前还是确认后。
- 如果确认后请求改变场景结构，提示用户回到 Scene Plan 阶段重新确认。
- 召回相关原文证据和现有剧本片段。
- 生成新版本。
- 运行 Validation Agent。
- 运行 Schema Validator。
- 保存 checkpoint。
- 展示修改前后差异。

## 17. 上下文与记忆治理

### 17.1 基本口径

系统不删除、不截断用户上传的原始材料。

系统采用统一 session 记忆库，但不同任务使用不同 Prompt 注入包。生成 Scene Plan、生成剧本、用户对话修改三类场景不应使用同一套上下文。

治理对象是派生记忆和 Prompt 注入内容，不是原始材料。

### 17.2 统一 session 记忆库

同一个项目 session 中保存以下内容：

- 小说原文。
- 转换后的 Markdown。
- 章节列表。
- 段落编号。
- 章节结构化摘要。
- 原文证据索引。
- Story Bible。
- Scene Plan。
- 当前剧本版本。
- 历史剧本版本。
- 用户修改历史。
- 用户确认过的决策。
- 风格参考剧本。
- Style Profile。
- checkpoint。
- execution log。

这些内容可以统一存储，但每次 LLM 调用不能全部写入 Prompt。

### 17.3 每次 Prompt 的固定注入包

每次创作或校验类 LLM 调用都必须构造 Prompt 注入包。注入包至少包含：

- 当前任务阶段。
- 当前任务目标。
- 当前处理范围。
- 用户最新指令。
- 用户已确认的关键决策。
- 当前允许变更的边界。
- 本轮需要使用的相关记忆。
- 本轮需要使用的相关原文证据。
- 输出格式和 Schema 约束。
- 本轮不能违反的禁止规则。

Prompt 注入包必须按 run_id 保存到开发者运行日志，并在数据库中记录摘要和关联关系。

### 17.4 生成 Scene Plan 的记忆注入

生成或重新生成 Scene Plan 时，Prompt 重点注入：

- 已确认的章节范围。
- 章节结构化摘要。
- 原文证据索引中的关键事件、地点、冲突、伏笔。
- Story Bible。
- Style Profile。
- 用户选择的剧本类型。
- 用户选择的改编策略。
- 用户对 Scene Plan 的最新反馈。

不默认注入：

- 全部小说原文。
- 全部参考剧本原文。
- 已生成剧本全文。
- 完整 execution log。
- 与场景规划无关的低相关证据。

Scene Plan 阶段的目标是决定“哪些小说内容应变成哪些剧本场景”。

### 17.5 生成剧本的记忆注入

逐场生成剧本时，Prompt 重点注入：

- 已确认的 Scene Plan。
- 当前场景的场景编号、场景目标、核心冲突和出场人物。
- 当前场景的来源证据。
- 当前场景相关段落摘录。
- Story Bible 中与当前场景相关的人物、关系、地点和时间线。
- Style Profile 中与当前场景相关的风格规则。
- 前后相邻场景摘要。
- JSON/YAML Schema 约束。

不默认注入：

- 全部章节摘要。
- 全部原文证据索引。
- 全部小说原文。
- 全部风格参考剧本。
- 全部历史版本。

剧本生成阶段的目标是把已确认 Scene Plan 中的单个场景写成准确、连贯、可校验的剧本内容。

### 17.6 用户对话修改的记忆注入

用户通过对话修改剧本时，Prompt 重点注入：

- 用户当前指令。
- 当前剧本版本。
- 用户指定的修改范围。
- 已确认的 Scene Plan。
- 当前范围相关原文证据。
- 当前范围相关剧本片段。
- 相关历史修改记录。
- 用户偏好。
- Style Profile。
- 当前版本与上一版本的差异。
- 是否允许改变 Scene Plan。

如果 Scene Plan 已确认，用户对话默认只修改剧本内容，不隐式改变场景数量、顺序、来源证据和核心冲突。

如果用户明确要求删场、合并场、重排场景或重新规划场景，系统应提示该请求属于 Scene Plan 变更，需要回到 Scene Plan 阶段重新确认。

用户对话修改阶段的目标是判断“用户要改哪里、是否允许改、改完是否仍符合已确认规划”。

### 17.7 检索、注入与压缩规则

原始材料完整保存，不做删除式裁剪。

第一版的“召回”指从当前项目 session 已保存材料中取出相关内容进入 Prompt，不默认等同于向量检索或复杂 RAG。

支持三类召回方式：

#### ID 召回

根据稳定编号直接取回材料。

适用场景：

- Scene Plan 中有来源证据编号，需要取回对应原文证据。
- 剧本场景中有段落编号，需要取回对应原文摘录。
- Validation Agent 发现某个证据缺失，需要取回证据详情。

ID 召回是第一版最重要的召回方式。

#### 范围召回

根据当前任务范围取回相关材料。

适用场景：

- 用户要求修改某一场。
- 用户要求修改某个角色的对白。
- 用户要求检查某一章的改编效果。

范围召回应取回当前范围内的剧本片段、相关证据、相关 Story Bible 片段、前后相邻场景摘要。

#### 关键词和全文检索召回

根据人物名、地点、证据类型、段落文本或用户指令关键词检索相关材料。

适用场景：

- 用户用自然语言描述修改对象，但没有给出明确场景编号。
- 用户提到某个人物关系、地点或事件。
- 系统需要查找与某个冲突、伏笔或对白相关的材料。

第一版可以使用 PostgreSQL 文本检索能力完成，不强制使用向量数据库。

派生记忆按任务相关性和重要性决定是否注入 Prompt：

| 记忆类型 | 处理方式 |
| --- | --- |
| 当前任务必需记忆 | 直接注入 Prompt |
| 高相关记忆 | 摘要后注入，必要时附证据 |
| 中相关记忆 | 通过检索召回，不主动注入 |
| 低相关记忆 | 仅保留在 session，不进入本轮 Prompt |
| 冲突或过期记忆 | 标记失效，不参与生成 |

判断依据：

- 是否影响当前任务目标。
- 是否属于用户已确认决策。
- 是否与当前章节、场景、角色或对白相关。
- 是否绑定关键事件、伏笔或核心冲突。
- 是否被 Validation Agent 标记为可靠或有风险。
- 是否与用户最新修改冲突。

## 18. 开发者运行日志与透明化要求

系统必须提供开发者运行日志面板。

开发者能看到：

- run_id。
- 当前阶段。
- 本轮 Prompt 拼接结果。
- 本轮 Prompt 注入包类型。
- 本轮注入的章节摘要。
- 本轮召回的原文证据。
- 本轮使用的 Style Profile 规则。
- 本轮未注入但可检索的相关材料摘要。
- LLM 模型名。
- LLM 输出。
- 工具调用名称。
- 工具调用参数。
- 工具返回结果。
- Validation Agent 报告。
- Schema Validator 结果。
- 修复原因。
- checkpoint ID。

普通用户界面默认展示生成阶段、校验结论、引用证据和版本变化，不默认展示完整 Prompt、工具参数和工具返回结果。

开发者运行日志必须写入 `developer_runs/{run_id}/`，并在数据库中保存 run 摘要、路径和关联关系。

日志允许折叠展示，但开发者不能完全隐藏。

## 19. 调用限制与循环限制

### 19.1 项目级限制

- MVP 推荐处理 3-5 章。
- 单章建议 2,000-8,000 字。
- 单个 session 推荐总字数不超过 50,000 字。
- 单个 session Agent 主循环最多 12 轮。
- 单个 session 工具调用总数默认最多 50 次。

### 19.2 单场限制

每个场景最多执行：

```text
生成
→ 校验
→ 修复
→ 复检
```

限制：

- Script Generation Worker 生成：最多 1 次。
- Validation Agent 校验：最多 2 次。
- 自动修复：最多 2 次。
- Schema Validator：最多 3 次。

仍失败时标记为“需要人工复核”，不得无限循环。

### 19.3 对话修改限制

每次用户对话修改默认只处理一个明确范围。

一次修改最多触发：

- 1 次上下文召回。
- 1 次生成。
- 1 次 Validation Agent 校验。
- 1 次自动修复。
- 1 次复检。

## 20. Session、Checkpoint 与版本

### 20.1 项目对话

用户可以选择新建项目对话。每个项目对话是一次独立的小说改编工作空间。

第一版默认关系：

```text
一个项目 project
→ 一个剧本生成 script_generation
→ 一个主对话 conversation
→ 一个 active session
```

用户新建项目时，可以上传不同小说，并为该小说生成独立剧本。不同项目之间默认隔离。

数据模型需要保留扩展能力：未来允许一个项目下的同一个剧本生成打开多个新对话，用于围绕同一剧本进行不同方向的继续修改或讨论。

新建项目对话时，系统应创建：

- project ID。
- script_generation ID。
- primary conversation ID。
- session ID。
- 空的对话上下文。
- 空的文件集合。
- 空的中间产物集合。
- 初始 checkpoint。

不同项目对话之间默认隔离：

- 不共享小说原文。
- 不共享 Story Bible。
- 不共享 Scene Plan。
- 不共享当前剧本版本。
- 不共享 execution log。
- 不默认注入其他项目的记忆。

如果用户希望复用其他项目的风格参考或剧本片段，必须显式导入。

### 20.2 Session

第一版每个项目创建一个 active session。该 session 从项目对话创建开始，贯穿上传、分析、生成、修改、导出和断点恢复。

session 状态以对话记录为主线保存。用户从最初上传文件、提出要求，到最终导出剧本，所有关键输入、系统响应、生成结果、修改请求和 checkpoint 都应能按对话顺序回放。

未来如果一个项目下开启多个对话，每个 conversation 可以拥有自己的 session 记录，但共享同一个项目和剧本生成的基础数据。

session 保存：

- 用户输入。
- 对话消息记录。
- 中间产物。
- 当前版本。
- Style Profile。
- Execution Log。
- checkpoint 列表。
- run_id 列表。

### 20.3 Checkpoint

以下节点必须保存 checkpoint：

- 文件上传完成。
- Markdown 转换完成。
- 章节确认完成。
- 段落编号完成。
- 章节摘要完成。
- 证据索引完成。
- Story Bible 完成。
- Style Profile 完成。
- Scene Plan 确认完成。
- 每个场景生成完成。
- 每次对话修改完成。
- 最终导出完成。

### 20.4 断点恢复

用户重新打开 session 后，可以：

- 查看上次进度。
- 从最近 checkpoint 继续。
- 回退到指定 checkpoint。
- 查看历史版本差异。

## 21. 输出需求

### 21.1 内部输出

内部核心输出是 JSON。

JSON 用于：

- 前端编辑。
- Schema 校验。
- 版本 diff。
- 断点恢复。
- 转换为其他格式。

### 21.2 必选输出

YAML 必须输出。

右侧成果区默认展示 YAML。YAML 必须符合预定义 Schema，并包含：

- 剧本标题。
- 人物表。
- 场景列表。
- 场景信息。
- 动作。
- 对白。
- 旁白。

用户最终导出的 YAML 默认不包含来源证据编号、段落编号或内部追溯 ID。

系统内部必须保留 traceability_index，用于把剧本场景、内容块和原文证据建立映射关系。页面中用户点击来源证据时，系统通过 traceability_index 弹出对应章节、段落和原文摘录。

### 21.3 可选输出

用户可选导出：

- Markdown。
- DOC/DOCX。
- PDF。
- TXT。
- JSON。

导出层只转换用户可见的剧本内容，不改变剧本正文，不携带内部来源证据编号、段落编号或 traceability_index。

导出时用户选择目标格式。可选格式不在主界面作为并列预览 tab 展示。

## 22. Schema 文档

系统必须输出 YAML Schema 说明文档。

文档应包含：

- Schema 设计目标。
- 顶层结构说明。
- 人物表字段说明。
- 场景列表字段说明。
- 内容块字段说明。
- 内部追溯索引说明。
- 为什么按场景组织。
- 为什么最终导出文件不包含来源证据编号。
- 页面如何通过 traceability_index 追溯原文证据。
- 校验规则说明。
- 可扩展字段说明。

## 23. 前端页面

第一版采用单工作台界面，整体为左中右三栏结构，不以多页面跳转为主要交互方式。

三栏职责：

- 左侧：项目对话入口、流程进度、历史 run。
- 中间：用户与 Agent 的对话、当前操作按钮、生成状态。
- 右侧：Scene Plan 和最终剧本成果查看区。

### 23.1 项目对话入口页

应包含：

- 新建项目对话。
- 打开已有项目对话。
- 查看项目对话状态。
- 查看最近 checkpoint。
- 查看最近修改时间。

### 23.2 三栏工作台

左侧栏应包含：

- 新建项目对话。
- 打开已有项目对话。
- 当前项目流程进度。
- 历史 run 列表。

中间栏应包含：

- 用户与 Agent 的对话记录。
- 当前阶段说明。
- 当前可执行操作。
- 对话输入框。

右侧栏应包含：

- Scene Plan 查看区。
- YAML 剧本查看区。
- 当前选中场景详情。
- 导出入口。

右侧栏是成果区，不默认放证据、Story Bible、Style Profile、校验报告或开发者 run 日志。

### 23.3 右侧成果区

应包含：

- Scene Plan 确认前：展示 Scene Plan。
- Scene Plan 确认后：展示生成出的 YAML 剧本。
- 页面默认不展示 `[剧本文本] [YAML] [JSON]` 并列预览 tab。
- 最终界面展示以 YAML 为主。
- 不同导出格式只在导出动作中选择。
- 来源证据以可点击标记或按钮形式展示在页面层。
- 用户点击来源证据后，弹窗展示对应章节、段落编号和原文内容。
- 用户最终导出的文件不包含来源证据编号。

## 24. 验收标准

第一版通过验收需要满足：

1. 能新建项目对话。
2. 能上传至少 3 个章节的小说文本。
3. 能将输入文件统一转换为 Markdown。
4. 能识别并确认章节。
5. 能生成稳定段落编号。
6. 能生成章节结构化摘要。
7. 能生成原文证据索引。
8. 能生成 Story Bible。
9. 能上传风格参考剧本或选择内置风格，并生成 Style Profile。
10. 能生成 Scene Plan，并支持用户确认。
11. 能逐场生成剧本 JSON。
12. 能导出合法 YAML。
13. 能输出 YAML Schema 说明文档。
14. 能进行 Validation Agent 校验。
15. 能进行程序化 Schema 校验。
16. 能为开发者保存并展示 Prompt、工具调用、工具返回和校验报告。
17. 能保存 session 和 checkpoint。
18. 能从 checkpoint 恢复。
19. 能通过对话修改至少一个场景，并保存新版本。

## 25. 风险与约束

### 25.1 长文本风险

风险：章节过长会导致上下文成本高、摘要遗漏、生成偏离。

应对：分块摘要、证据索引、逐场生成、覆盖率检查。

### 25.2 PDF 解析风险

风险：PDF 排版复杂或扫描版无法准确提取。

应对：第一版只支持文本型 PDF，推荐用户上传 TXT、DOCX 或 Markdown。

### 25.3 风格误判风险

风险：Style Profile 可能过度概括用户风格。

应对：展示提取结果，允许用户编辑风格规则。

### 25.4 校验误判风险

风险：Validation Agent 也可能遗漏问题。

应对：结合程序化 Schema Validator，并保留人工复核状态。

### 25.5 自动修复循环风险

风险：模型反复修复失败。

应对：严格限制修复次数，失败后标记人工复核。

## 26. 第一版成功标准

第一版成功不要求生成专业成品剧本，而要求证明系统具备：

- 长文本结构化处理能力。
- 原文证据追溯能力。
- 剧本 YAML 生成能力。
- 校验与修复能力。
- 对话式修改能力。
- 风格参考能力。
- session 保存与断点恢复能力。
- 开发者运行日志追踪能力。

一句话总结：

NovelScript AI 是一个由主控 Agent 编排多 Worker，并由独立 Validation Agent 校验的小说剧本改编工作台。
