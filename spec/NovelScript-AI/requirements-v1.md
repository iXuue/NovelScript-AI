# NovelScript AI 第一版产品需求

版本：v0.1 MVP  
项目名称：NovelScript AI  
定位：可追溯、可校验、可对话修改的 AI 小说剧本改编工作台

## 1. 产品目标

NovelScript AI 面向小说作者和内容创作者，帮助用户把 3-5 个章节的小说文本转换为结构化剧本初稿。

第一版重点不是“让 AI 直接写剧本”，而是建立一条稳定的长文本改编流程：

```text
输入文件 → Markdown → 章节确认 → 段落编号
                         ↓
        ┌────────────────┴────────────────┐
        ↓                                 ↓
章节摘要 / 原文证据索引             风格来源处理
        ↓                                 ↓
  Story Bible                      Style Profile
        └────────────────┬────────────────┘
                         ↓
                    Scene Plan
                         ↓
                  用户确认 Scene Plan
                         ↓
                    逐场生成剧本
                         ↓
                    校验与修复
                         ↓
         必选导出 YAML，可选导出其他格式
```

执行规则：章节摘要、原文证据索引和风格来源处理可以并行执行；风格来源处理包括上传参考剧本分析、用户文字风格描述解析或内置风格生成。Story Bible 需要在章节摘要和证据索引完成后生成；Scene Plan 需要在 Story Bible、Style Profile 和相关证据完成后生成。

## 2. 核心原则

1. 原始材料完整保存，不做删除式裁剪。
2. 长文本通过摘要、结构化记忆和证据索引进入生成流程。
3. 生成过程必须可追踪，开发者能查看 Prompt、工具调用、工具返回和校验报告。
4. 剧本生成过程必须可追溯，内部数据需要绑定原文证据，但用户最终导出文件默认不包含来源证据编号。
5. 内部优先使用 JSON，最终必须输出符合 Schema 的 YAML。
6. 用户可以通过对话持续修改剧本，而不是只能一次性生成。

## 3. MVP 范围

### 3.1 必须支持

- 用户可以新建项目。
- 上传小说文本并转换为 Markdown。
- 支持 3-5 个章节的小说改编。
- 自动识别章节，并允许用户确认章节划分。
- 给章节正文建立稳定段落编号。
- 生成章节结构化摘要。
- 生成原文证据索引。
- 生成 Story Bible。
- 支持三种风格来源：选择系统内置风格、上传过往剧本作为风格参考、输入文字描述目标风格。
- 生成 Style Profile。
- 生成 Scene Plan，并允许用户通过对话确认或调整；第一版不支持用户手动编辑 Scene Plan 字段。
- 逐场生成结构化剧本 JSON。
- 使用 Validation Agent 校验内容质量。
- 使用程序化 Schema Validator 校验结构。
- 修复失败内容并限制重试次数。
- 必选导出 YAML。
- 可选导出 Markdown、DOCX、DOC、PDF、TXT、用户干净 JSON。
- 保存 session、checkpoint、execution log 和 developer run logs。
- 支持断点恢复。

### 3.2 暂不支持

- 整本百万字小说一次性改编。
- 扫描版 PDF OCR。
- 图片、漫画、视频输入。
- 多人协作编辑。
- 真正训练或微调用户专属模型。
- 专业影视分镜生成。
- 配音、视频、音频生成。
- 复杂出版级排版。

## 4. 用户角色

### 4.1 小说作者

希望将已有小说改编为剧本初稿，降低改编门槛。

### 4.2 内容创作者

希望快速生成短剧、网剧或影视剧本雏形，再继续人工修改。

## 5. 输入需求

### 5.1 支持格式

第一版支持：

- Markdown：`.md`
- 文本文档：`.txt`
- Word 文档：支持 `.docx`，并通过 LibreOffice 转换支持 `.doc`。
- PDF：`.pdf`

PDF 限制：仅保证普通文本型 PDF 的正文提取，不保证扫描版 PDF、图片型 PDF 或复杂排版 PDF 的准确解析。

### 5.2 输入转换规则

所有输入文件先进入 Input Adapter，统一转换为 Markdown。

```text
docx/txt/pdf/md
→ Input Adapter
→ Markdown
→ Markdown Normalizer
→ normalized_chapters.json
```

`.doc` 通过 LibreOffice 转换为 `.docx` 后进入正文提取流程。

转换时需要：

- 保留段落边界。
- 保留章节标题。
- 清理明显无意义空行。
- 不删除短对白段落。
- 不直接修改原始文件。
- 保存原始文件和转换后的 Markdown。

### 5.3 上传模式

支持两种小说上传模式：

1. 单文件多章。
2. 多文件单章。

多文件上传时需要自然排序，并允许用户手动调整顺序。

## 6. 章节与段落处理

### 6.1 章节识别

系统应识别常见章节标题：

- 第一章
- 第1章
- Chapter 1
- CHAPTER 1
- 一、标题
- 1. 标题
- 序章、楔子、番外

识别后进入章节确认面板，用户可以：

- 确认章节。
- 修改章节标题。
- 调整章节顺序。
- 重新识别。

### 6.2 段落编号

确认章节后，系统为每个段落生成稳定编号。

示例：

```text
CH001_P001
CH001_P002
CH002_P001
```

段落编号用于：

- 原文证据索引。
- 场景规划。
- 剧本来源追踪。
- 校验报告。
- 断点恢复。

段落编号生成后不得随意重排。

## 7. 逻辑产物清单

第一版至少定义以下逻辑产物：

```text
raw_files/
normalized_markdown.md
normalized_chapters.json
paragraph_index.json
chapter_summaries.json
evidence_index.json
story_bible.json
style_profile.json
scene_plan.json
script_scenes/*.json
final_script.json
final_script.yaml
traceability_index.json
schema_document.md
adaptation_report.json
execution_log.jsonl
developer_runs/
checkpoints/
```

这些名称表示业务流程中的逻辑产物，不代表业务读取必须从文件系统读取。

业务产物默认存 PostgreSQL。文件系统只保存原始文件、导出文件和开发者调试快照。若数据库记录与调试快照不一致，以数据库记录为准。

业务读取默认走 PostgreSQL。文件系统中的 JSON、YAML、JSONL 如果属于调试快照，不参与业务读取。

## 8. 技术栈与数据存储

### 8.1 技术栈

第一版技术栈约束：

- 后端语言：Python。
- 运行环境：Docker。
- 主数据库：PostgreSQL。
- 结构化业务产物：优先使用 PostgreSQL JSONB 存储。
- 原始文件、导出文件和开发者调试快照：使用文件存储保存，数据库保存元数据、索引和追溯关系。

### 8.2 PostgreSQL 存储范围

PostgreSQL 至少需要保存：

- 项目。
- session。
- 对话 conversation。
- 剧本生成 script generation。
- 上传文件元数据。
- 章节。
- 段落编号。
- 原文证据。
- Story Bible。
- Style Profile。
- 风格来源类型。
- 用户文字风格描述。
- 风格参考文件关联。
- Scene Plan。
- 剧本版本。
- checkpoint。
- run 记录。
- execution log。
- 导出记录。

章节摘要、Story Bible、Style Profile、Scene Plan、剧本 JSON、校验报告、checkpoint snapshot 和 run 摘要可以使用 JSONB 字段保存。

PostgreSQL 是 session、checkpoint、逻辑业务产物、剧本版本、run 记录和追溯关系的权威数据源。文件系统中的 JSON、YAML 或 JSONL 文件只作为原始材料、导出结果或开发者调试快照使用。

### 8.3 原始文件追溯保存

用户上传的原始文件必须保存，不能只保存解析后的文本。

原始文件包括：

- 小说原文件。
- 作为风格参考上传的历史剧本。
- 导出的 YAML、Markdown、DOCX、DOC、PDF、TXT、用户干净 JSON 文件。

数据库需要保存每个文件的元数据：

- 文件 ID。
- 所属 project ID。
- 所属 session ID。
- 原始文件名。
- 文件类型。
- 文件大小。
- 文件 hash。
- 存储路径。
- 上传时间。
- 转换状态。
- 对应的 Markdown 文件 ID。
- 是否作为风格参考。
- 所属风格来源记录 ID。

后续所有章节、段落、证据、剧本场景和导出结果都应能追溯到对应原始文件。

### 8.4 开发者 run 日志文件夹

Prompt、工具调用参数、工具返回结果和模型输出仅供开发人员排查问题，不写入普通用户前端界面。

每次用户触发一次完整任务时，系统必须创建一个独立 `run_id`。完整任务包括初始生成、重新生成 Scene Plan、确认后逐场生成、用户对话修改、用户手动触发校验、用户重新导出等。

自动解析、自动校验、自动修复、Schema Validator、导出转换等不单独创建新的 `run_id`，而是作为同一 `run_id` 下的 `run_step_id` 记录。只有用户再次发起新的生成、修改、校验或导出请求时，才创建新的 `run_id`。

每个 `run_id` 对应一个独立开发者日志目录：

```text
developer_runs/{run_id}/
  run_meta.json
  run_steps.jsonl
  prompt.md
  prompt_injection_package.json
  model_output.json
  tool_calls.jsonl
  tool_results.jsonl
  validation_report.json
```

数据库需要保存 `run_id` 与 project、script_generation、conversation、session、checkpoint 的关联关系。

数据库需要保存 `run_step_id` 与 run、任务阶段、LLM 调用、工具调用、校验结果和 checkpoint 的关联关系。

开发者 run 日志需要避免保存 API key、数据库连接串、系统密钥和其他敏感凭据。日志可能包含小说原文、用户历史剧本或用户风格描述，因此必须遵守：

- 仅本地开发环境写入。
- 不提交到 Git。
- 不进入用户导出文件。
- 不同步到普通用户前端。
- 默认只保留最近 7 天或最近 50 个 run，以先达到者为准。
- 用户删除项目时，应同步删除该项目关联的本地开发日志。
- 能用摘要定位问题时，优先保存摘要；确需保存完整 Prompt、模型输出或工具返回时，只能保存在本地开发日志目录。

### 8.5 Docker 环境要求

Docker 环境至少包含：

- Python 后端服务。
- PostgreSQL 数据库服务。
- 文件存储挂载目录。

第一版可以使用本地 Docker Compose 管理后端和数据库。数据库数据目录和文件存储目录必须持久化挂载，避免容器重启后丢失 session、checkpoint 和原始文件。

## 9. 章节结构化摘要

章节摘要不是普通摘要，而是服务剧本改编的结构化信息。

每章摘要应包含：

- 一句话摘要。
- 详细摘要。
- 本章叙事功能。
- 本章改编价值。
- 主要人物。
- 人物关系变化。
- 重要地点。
- 关键剧情点。
- 戏剧冲突。
- 情绪节点。
- 对白候选。
- 视觉元素。
- 伏笔线索。
- 适合改编的场景。
- 改编建议。
- 可能遗漏风险。

如果章节过长，先按段落分块生成 block summary，再合并为 chapter summary。

## 10. 原文证据索引

系统必须建立 evidence_index，用于减少遗漏和幻觉。

每条证据应包含：

- 证据编号。
- 来源章节。
- 相关段落编号。
- 原文摘录。
- 证据类型。
- 证据说明。
- 关联人物。
- 关联地点。
- 关联剧情点。
- 重要性。
- 是否必须保留。

支持的证据类型：

- 关键事件。
- 对白。
- 人物目标。
- 人物关系。
- 心理描写。
- 视觉元素。
- 线索。
- 伏笔。
- 冲突。
- 地点描写。

关键剧情点如果没有证据支撑，应在校验报告中标记为风险。

## 11. Story Bible

Story Bible 是项目级故事记忆，用于保持人物、关系、地点、时间线和伏笔一致。

必须包含：

- 故事标题。
- 故事类型。
- 整体基调。
- 一句话故事。
- 核心主题。
- 主要人物表。
- 人物关系表。
- 重要地点表。
- 剧情时间线。
- 主线冲突。
- 伏笔与线索。

Story Bible 不直接写剧本，只作为生成约束和校验依据。

## 12. 风格参考与 Style Profile

### 12.1 风格来源

用户创建项目后、发起生成前需要选择一种风格来源：

1. 选择系统内置风格类型。
2. 上传过往剧本作为风格参考，让 AI 提炼风格。
3. 输入文字描述目标风格，让 AI 解析为 Style Profile。

三种方式第一版互斥，不同时启用。同一轮生成请求中只能存在一种风格来源。

第一版最多支持上传 3 份参考剧本。

如果用户上传参考剧本，系统从参考剧本中提取 Style Profile。

如果用户输入文字描述目标风格，系统从用户描述中解析 Style Profile。

如果用户选择系统内置风格类型，系统根据所选风格对应的固定系统提示词生成默认 Style Profile。

系统不会训练模型，而是在当前 session 中生成或提取 Style Profile。

### 12.2 支持风格类型

第一版支持以下风格类型：

- 现实主义。
- 悬疑/惊悚。
- 爱情/情感。
- 喜剧。
- 短剧/网剧。

### 12.3 Style Profile 内容

Style Profile 应包含：

- 剧本类型。
- 对白密度。
- 场景长度。
- 节奏快慢。
- 角色说话风格。
- 冲突设计方式。
- 旁白比例。
- 动作描写比例。
- 常见转场方式。
- 用户偏好。

初始 Scene Plan 和初始剧本生成时，系统只注入相关风格规则和必要参考片段，不把全部参考剧本直接塞入 Prompt。用户文字风格描述需要保存为风格来源材料，但进入初始生成 Prompt 时仍应先转化为 Style Profile。

用户确认 Scene Plan 后锁定风格来源。Scene Plan 确认前允许清空当前风格输入或上传文件，并重新选择风格来源；重新选择后必须重新生成 Style Profile 和 Scene Plan。

用户不允许直接编辑 Style Profile 或风格规则。用户输入的文字风格描述只是生成 Style Profile 的输入材料，不等于开放编辑系统风格规则。Style Profile 只能由上传参考剧本分析、用户文字风格描述解析或内置风格生成得到。

系统内置风格只用于初始 Style Profile、Scene Plan 和剧本生成。Scene Plan 确认后，用户后续对话中提到“更悬疑”“更口语化”“更像短剧”等风格词时，系统应把它当作普通修改意见拼接进本轮 Prompt。

Scene Plan 确认后的风格词路由规则：

- 不触发内置风格选择。
- 不切换风格来源。
- 不重跑 Style Profile Worker。
- 不修改 Style Profile 或系统风格规则。
- 只影响本次对话修改 run 中被用户指定的修改范围。

## 13. Scene Plan

系统生成剧本前必须先生成 Scene Plan。

Scene Plan 用于决定小说内容如何拆成剧本场景。小说章节和剧本场景不是一一对应关系。

每个场景应包含：

- 场景编号。
- 场景标题。
- 场景顺序。
- 来源章节。
- 来源证据编号。
- 地点。
- 时间。
- 内外景。
- 出场人物。
- 场景功能。
- 核心冲突。
- 必须覆盖的剧情。
- 必须保留的对白。
- 必须保留的视觉元素。
- 必须保留的伏笔。
- 改编说明。

用户需要在生成剧本前确认 Scene Plan。

Scene Plan 确认必须是用户显式动作，不能由系统自动默认确认。第一版支持两种确认方式：

- 用户点击 `确认 Scene Plan` 按钮。
- 用户在对话中明确表达确认，例如“确认这个 Scene Plan”“按这个规划继续生成”。

系统识别到确认后，需要保存 Scene Plan 确认 checkpoint，锁定风格来源，并进入逐场生成剧本阶段。

确认前，Scene Plan 属于可协商阶段。用户可以通过对话提出调整要求，例如“我不喜欢这个规划，我想要节奏更快”“减少感情戏，增加悬疑感”“把前三章压缩成更少的场景”。系统应根据用户反馈重新生成或局部调整 Scene Plan，并保存新的 Scene Plan 版本。

第一版不提供手动编辑 Scene Plan 字段的入口。用户对 Scene Plan 的调整只能通过对话提出。

Scene Plan 确认后进入剧本生成阶段。此后用户仍可通过对话修改剧本，但默认只修改已确认 Scene Plan 下的剧本内容，不隐式改变 Scene Plan 的场景数量、顺序、来源证据和核心冲突。

如果用户在剧本生成后明确要求改变场景结构，例如“删掉第二场”“把这两场合并”“重新规划场景”，系统应提示这属于 Scene Plan 变更，需要回到 Scene Plan 阶段并重新确认后再继续生成。

## 14. Agent 架构

第一版采用主控 Agent 编排的多 Worker 工作流，并保留独立 Validation Agent：

```text
Orchestrator Agent
+ Specialized Workers
+ Validation Agent
+ Programmatic Validators
```

### 14.1 Orchestrator Agent

负责：

- 编排整体流程。
- 管理 session、checkpoint 和调用次数。
- 组织每次 LLM 调用的 Prompt 注入包。
- 调度 Specialized Workers。
- 根据用户对话修改剧本。
- 根据校验报告执行修复。

### 14.2 Specialized Workers

Specialized Workers 是受控任务执行单元，可以拥有独立 Prompt、工具和输出格式，但不自主循环、不自行规划、不越权调用其他 Worker。

第一版包含：

- Chapter Summary Worker：生成章节结构化摘要。
- Evidence Extraction Worker：生成原文证据索引。
- Style Profile Worker：根据上传参考剧本、用户文字风格描述或内置风格生成 Style Profile。
- Story Bible Worker：汇总人物、关系、地点、时间线和伏笔。
- Scene Plan Worker：生成或调整 Scene Plan。
- Script Generation Worker：逐场生成剧本 JSON。

### 14.3 Validation Agent

使用不同模型，只负责检查，不负责自由创作。

负责：

- 检查剧情覆盖率。
- 检查角色一致性。
- 检查场景是否符合 Scene Plan。
- 检查是否遗漏关键证据。
- 检查是否出现严重幻觉。
- 检查是否符合 Style Profile。
- 输出结构化校验报告。

Validation Agent 不直接改剧本。

### 14.4 Programmatic Validators

负责确定性校验：

- 内部 JSON Schema 校验。
- 用户导出 YAML / 用户干净 JSON Schema 校验。
- traceability_index 校验。
- 必填字段校验。
- 编号唯一性校验。
- 内部引用 ID 是否存在。
- YAML 是否可解析。

内部 JSON Schema 允许包含 `source_evidence_ids`、段落 ID、`content_block_id` 和追溯映射字段。

用户导出 YAML / 用户干净 JSON Schema 不包含来源证据编号、段落编号、`content_block_id` 或其他内部追溯字段。

traceability_index 需要单独校验，用于确认内部剧本 JSON 的内容块、来源证据和原文段落之间的映射关系有效。

## 15. 剧本生成

系统逐场生成剧本，不一次性生成完整剧本。

单场生成上下文应包含：

- 当前 Scene Plan。
- 相关章节摘要。
- 相关原文证据。
- Story Bible。
- Style Profile。
- Schema 约束。
- 用户当前修改要求。

内部输出为 JSON，最后转换为用户可查看的 YAML。

单场 JSON 至少包含：

- 场景编号。
- 场景标题。
- 来源章节。
- 来源证据。
- content_block_id。
- 场景信息。
- 出场人物。
- 场景目的。
- 核心冲突。
- 内容块列表。

内容块类型：

- 动作。
- 对白。
- 旁白。
- 转场。
- 注释。

对白必须包含说话人。

内部剧本 JSON 每个内容块必须包含稳定的 `content_block_id`。右侧页面通过 `content_block_id` 查询 traceability_index，再定位对应来源证据和原文段落。

来源证据 ID 保留在内部 JSON、traceability_index 和页面追溯层中，用于校验、回放和弹窗查看原文，不默认进入用户最终导出文件。导出时必须移除 `content_block_id`、来源证据 ID、段落 ID 和其他内部追溯字段。

## 16. 对话式修改

用户可以通过自然语言持续修改剧本。

对话式修改分为两个阶段：

1. Scene Plan 确认前：用户对话修改的是场景规划，系统可以重新生成或调整 Scene Plan。
2. Scene Plan 确认后：用户对话修改的是剧本内容，系统默认沿用已确认的 Scene Plan，不主动改变场景结构。

支持修改范围：

- 全剧。
- 某一章。
- 某一场。
- 某个角色。
- 某段对白。
- 某段表达风格。这里的风格修改只作为普通对话修改意见，不改变系统风格来源、Style Profile 或内置风格规则。

每次修改需要：

- 解析用户意图。
- 确定修改范围。
- 判断修改发生在 Scene Plan 确认前还是确认后。
- 如果确认后请求改变场景结构，提示用户回到 Scene Plan 阶段重新确认。
- 召回相关原文证据和现有剧本片段。
- 生成新版本。
- 运行 Validation Agent。
- 运行 Schema Validator。
- 保存 checkpoint。
- 展示修改前后差异。

## 17. 上下文与记忆治理

### 17.1 基本口径

系统不删除、不截断用户上传的原始材料。

系统采用统一 session 记忆库，但不同任务使用不同 Prompt 注入包。生成 Scene Plan、生成剧本、用户对话修改三类场景不应使用同一套上下文。

治理对象是派生记忆和 Prompt 注入内容，不是原始材料。

### 17.2 统一 session 记忆库

同一个项目 session 中保存以下内容：

- 小说原文。
- 转换后的 Markdown。
- 章节列表。
- 段落编号。
- 章节结构化摘要。
- 原文证据索引。
- Story Bible。
- Scene Plan。
- 当前剧本版本。
- 历史剧本版本。
- 用户修改历史。
- 用户确认过的决策。
- 风格来源材料。
- 风格参考剧本。
- Style Profile。
- checkpoint。
- execution log。

这些内容可以统一存储，但每次 LLM 调用不能全部写入 Prompt。

### 17.3 每次 Prompt 的固定注入包

每次创作或校验类 LLM 调用都必须构造 Prompt 注入包。注入包至少包含：

- 当前任务阶段。
- 当前任务目标。
- 当前处理范围。
- 用户最新指令。
- 用户已确认的关键决策。
- 当前允许变更的边界。
- 本轮需要使用的相关记忆。
- 本轮需要使用的相关原文证据。
- 输出格式和 Schema 约束。
- 本轮不能违反的禁止规则。

Prompt 注入包必须按 run_id 保存到开发者运行日志，并在数据库中记录摘要和关联关系。

### 17.4 生成 Scene Plan 的记忆注入

生成或重新生成 Scene Plan 时，Prompt 重点注入：

- 已确认的章节范围。
- 章节结构化摘要。
- 原文证据索引中的关键事件、地点、冲突、伏笔。
- Story Bible。
- Style Profile。
- 用户选择的剧本类型。
- 用户选择的改编策略。
- 用户对 Scene Plan 的最新反馈。

不默认注入：

- 全部小说原文。
- 全部参考剧本原文。
- 已生成剧本全文。
- 完整 execution log。
- 与场景规划无关的低相关证据。

Scene Plan 阶段的目标是决定“哪些小说内容应变成哪些剧本场景”。

### 17.5 生成剧本的记忆注入

逐场生成剧本时，Prompt 重点注入：

- 已确认的 Scene Plan。
- 当前场景的场景编号、场景目标、核心冲突和出场人物。
- 当前场景的来源证据。
- 当前场景相关段落摘录。
- Story Bible 中与当前场景相关的人物、关系、地点和时间线。
- Style Profile 中与当前场景相关的风格规则。
- 前后相邻场景摘要。
- JSON/YAML Schema 约束。

不默认注入：

- 全部章节摘要。
- 全部原文证据索引。
- 全部小说原文。
- 全部风格来源材料。
- 全部风格参考剧本。
- 全部历史版本。

剧本生成阶段的目标是把已确认 Scene Plan 中的单个场景写成准确、连贯、可校验的剧本内容。

### 17.6 用户对话修改的记忆注入

用户通过对话修改剧本时，Prompt 重点注入：

- 用户当前指令。
- 当前剧本版本。
- 用户指定的修改范围。
- 已确认的 Scene Plan。
- 当前范围相关原文证据。
- 当前范围相关剧本片段。
- 相关历史修改记录。
- 用户偏好。
- Style Profile。
- 当前版本与上一版本的差异。
- 是否允许改变 Scene Plan。

如果 Scene Plan 已确认，用户对话默认只修改剧本内容，不隐式改变场景数量、顺序、来源证据和核心冲突。

如果用户明确要求删场、合并场、重排场景或重新规划场景，系统应提示该请求属于 Scene Plan 变更，需要回到 Scene Plan 阶段重新确认。

用户对话修改阶段的目标是判断“用户要改哪里、是否允许改、改完是否仍符合已确认规划”。

### 17.7 检索、注入与压缩规则

原始材料完整保存，不做删除式裁剪。

第一版的“召回”指从当前项目 session 已保存材料中取出相关内容进入 Prompt，不默认等同于向量检索或复杂 RAG。

支持三类召回方式：

#### ID 召回

根据稳定编号直接取回材料。

适用场景：

- Scene Plan 中有来源证据编号，需要取回对应原文证据。
- 剧本场景中有段落编号，需要取回对应原文摘录。
- Validation Agent 发现某个证据缺失，需要取回证据详情。

ID 召回是第一版最重要的召回方式。

#### 范围召回

根据当前任务范围取回相关材料。

适用场景：

- 用户要求修改某一场。
- 用户要求修改某个角色的对白。
- 用户要求检查某一章的改编效果。

范围召回应取回当前范围内的剧本片段、相关证据、相关 Story Bible 片段、前后相邻场景摘要。

#### 关键词和全文检索召回

根据人物名、地点、证据类型、段落文本或用户指令关键词检索相关材料。

适用场景：

- 用户用自然语言描述修改对象，但没有给出明确场景编号。
- 用户提到某个人物关系、地点或事件。
- 系统需要查找与某个冲突、伏笔或对白相关的材料。

第一版可以使用 PostgreSQL 文本检索能力完成，不强制使用向量数据库。

派生记忆按任务相关性和重要性决定是否注入 Prompt：

| 记忆类型 | 处理方式 |
| --- | --- |
| 当前任务必需记忆 | 直接注入 Prompt |
| 高相关记忆 | 摘要后注入，必要时附证据 |
| 中相关记忆 | 通过检索召回，不主动注入 |
| 低相关记忆 | 仅保留在 session，不进入本轮 Prompt |
| 冲突或过期记忆 | 标记失效，不参与生成 |

判断依据：

- 是否影响当前任务目标。
- 是否属于用户已确认决策。
- 是否与当前章节、场景、角色或对白相关。
- 是否绑定关键事件、伏笔或核心冲突。
- 是否被 Validation Agent 标记为可靠或有风险。
- 是否与用户最新修改冲突。

## 18. 开发者运行日志与透明化要求

系统必须保存开发者运行日志。开发者运行日志仅本地开发可见，仅供开发人员在本地开发环境或本地日志目录查看；普通用户前端不显示开发者模式、不显示 run 日志。

开发者能看到：

- run_id。
- run_step_id。
- 当前阶段。
- 本轮 Prompt 拼接结果。
- 本轮 Prompt 注入包类型。
- 本轮注入的章节摘要。
- 本轮召回的原文证据。
- 本轮使用的 Style Profile 规则。
- 本轮未注入但可检索的相关材料摘要。
- LLM 模型名。
- LLM 输出。
- 工具调用名称。
- 工具调用参数。
- 工具返回结果。
- Validation Agent 报告。
- Schema Validator 结果。
- 修复原因。
- checkpoint ID。

普通用户界面默认展示生成阶段、校验结论、引用证据和版本变化，不展示完整 Prompt、工具参数、工具返回结果和 run 详情。

开发者运行日志必须写入 `developer_runs/{run_id}/`，并在数据库中保存 run 摘要、路径和关联关系。

开发者运行日志不进入普通用户前端，也不作为第一版线上用户功能。开发人员只能在本地开发环境或本地文件目录查看。

## 19. 调用限制与循环限制

### 19.1 预算口径

第一版必须把预算拆成以下层级，不使用单一“session 总次数”覆盖所有行为：

- project/session 总预算。
- 单 run 预算。
- 单 scene 预算。
- LLM 调用预算。
- 工具调用预算。

`run` 指一次用户触发的完整任务。`run_step` 指该任务内部的解析、生成、校验、修复、导出等步骤。自动校验和自动修复默认属于当前 run 的 run_step，不单独创建新 run。

工具调用指由 Orchestrator 或 Worker 显式调度的外部工具、解析器、转换器、检索器或校验器调用；普通数据库读写和前端读取不计入工具调用预算。

### 19.2 Project / Session 总预算

- MVP 推荐处理 3-5 章。
- 单章建议 2,000-8,000 字。
- 单个 session 推荐总字数不超过 50,000 字。
- 单个 project 默认一个 active session。
- 单个 session 主流程阶段必须按 checkpoint 推进，不允许无条件循环。
- project/session 总预算需要在配置中显式定义，至少包含总 LLM 调用次数和总工具调用次数。

MVP 默认预算：

- 单个 project/session 最多处理 5 章。
- 单个 project/session 小说正文总字数默认不超过 50,000 字。
- 单个 project/session 总 LLM 调用默认最多 120 次。
- 单个 project/session 总工具调用默认最多 200 次。
- 单个 project/session 同一时间只允许 1 个 active run。

### 19.3 单 Run 预算

每个 run 必须记录：

- run_id。
- 触发来源。
- 任务阶段。
- run_step 列表。
- LLM 调用次数。
- 工具调用次数。
- 是否触发自动修复。
- 是否达到预算上限。

单 run 达到预算上限后必须停止继续循环，保存当前结果和失败原因。

MVP 单 run 默认预算：

- 初始分析与 Scene Plan run：LLM 调用最多 60 次，工具调用最多 100 次。
- Scene Plan 重新生成 run：LLM 调用最多 12 次，工具调用最多 25 次。
- 逐场剧本生成 run：LLM 调用最多 80 次，工具调用最多 120 次，但仍受 project/session 总预算限制。
- 对话修改 run：LLM 调用最多 6 次，工具调用最多 15 次。
- 用户手动触发校验 run：LLM 调用最多 4 次，工具调用最多 10 次。
- 导出 run：LLM 调用最多 0 次，工具调用最多 10 次。

### 19.4 单 Scene 预算

每个场景最多执行：

```text
生成
→ 校验
→ 修复
→ 复检
```

限制：

- Script Generation Worker 生成：最多 1 次。
- Validation Agent 校验：最多 2 次。
- 自动修复：最多 2 次。
- Schema Validator：最多 3 次。

仍失败时标记为“需要人工复核”，不得无限循环。第一版不做前端人工复核工作台或人工复核入口，该状态只保存为内部状态、校验报告和开发者日志。

### 19.5 LLM 调用预算

LLM 调用预算必须按任务类型区分，至少覆盖：

- 章节摘要。
- 原文证据索引。
- Story Bible。
- Style Profile。
- Scene Plan。
- 单场剧本生成。
- Validation Agent 校验。
- 自动修复。
- 对话修改。

每次 LLM 调用必须关联 run_id、任务类型、输入摘要、输出摘要和预算消耗。

每次 LLM 调用还必须关联 `run_step_id`。如果某个步骤触发自动修复，修复调用仍计入同一个 run 的预算。

### 19.6 工具调用预算

工具调用预算必须按工具类型区分，至少覆盖：

- 输入文件转换。
- Markdown 规范化。
- 章节识别。
- 段落编号。
- PostgreSQL 文本检索。
- Schema Validator。
- 导出转换。

每次工具调用必须记录工具名、参数摘要、返回摘要、耗时、失败原因和所属 run_id。

每次工具调用还必须关联 `run_step_id`。导出转换、Schema Validator、文件转换和文本检索均计入工具调用预算；普通数据库读写和前端查询不计入。

### 19.7 对话修改限制

每次用户对话修改默认只处理一个明确范围。

一次修改最多触发：

- 1 次上下文召回。
- 1 次生成。
- 1 次 Validation Agent 校验。
- 1 次自动修复。
- 1 次复检。

## 20. Session、Checkpoint 与版本

### 20.1 项目与对话

统一术语：

- 项目 project：小说和剧本生成的工作空间。
- 对话 conversation：围绕项目的聊天记录。
- 第一版每个 project 默认一个 primary conversation。

用户可以选择新建项目。每个项目是一次独立的小说改编工作空间。

第一版默认关系：

```text
一个项目 project
→ 一个剧本生成 script_generation
→ 一个主对话 conversation
→ 一个 active session
```

用户新建项目时，可以上传不同小说，并为该小说生成独立剧本。不同项目之间默认隔离。

数据模型需要保留扩展能力：未来允许一个项目下的同一个剧本生成打开多个新对话，用于围绕同一剧本进行不同方向的继续修改或讨论。

新建项目时，系统应创建：

- project ID。
- script_generation ID。
- primary conversation ID。
- session ID。
- 空的对话上下文。
- 空的文件集合。
- 空的逻辑产物集合。
- 初始 checkpoint。

不同项目之间默认隔离：

- 不共享小说原文。
- 不共享 Story Bible。
- 不共享 Scene Plan。
- 不共享当前剧本版本。
- 不共享 execution log。
- 不默认注入其他项目的记忆。

如果用户希望复用其他项目的风格参考或剧本片段，必须显式导入。

### 20.2 Session

第一版每个项目创建一个 active session。该 session 从项目创建开始，贯穿上传、分析、生成、修改、导出和断点恢复。

session 状态以对话记录为主线保存。用户从最初上传文件、提出要求，到最终导出剧本，所有关键输入、系统响应、生成结果、修改请求和 checkpoint 都应能按对话顺序回放。

未来如果一个项目下开启多个对话，每个 conversation 可以拥有自己的 session 记录，但共享同一个项目和剧本生成的基础数据。

session 保存：

- 用户输入。
- 对话消息记录。
- 逻辑产物。
- 当前版本。
- Style Profile。
- Execution Log。
- checkpoint 列表。
- run_id 列表。

### 20.3 Checkpoint

以下节点必须保存 checkpoint：

- 文件上传完成。
- Markdown 转换完成。
- 章节确认完成。
- 段落编号完成。
- 章节摘要完成。
- 证据索引完成。
- Story Bible 完成。
- Style Profile 完成。
- Scene Plan 确认完成。
- 每个场景生成完成。
- 每次对话修改完成。
- 最终导出完成。

### 20.4 产物依赖与失效规则

系统必须保存逻辑产物之间的依赖关系。上游产物变化时，下游产物不得继续作为当前有效产物参与生成，必须标记为 stale，并保留为历史版本或调试材料。

默认失效规则：

- 原始小说文件重新上传：Markdown、章节、段落编号、章节摘要、证据索引、Story Bible、Scene Plan、剧本 JSON、traceability_index、YAML 预览和导出结果全部失效。
- Markdown 转换结果变化：章节、段落编号、章节摘要、证据索引、Story Bible、Scene Plan、剧本 JSON、traceability_index、YAML 预览和导出结果全部失效。
- 章节顺序、章节范围或章节正文变化：段落编号、章节摘要、证据索引、Story Bible、Scene Plan、剧本 JSON、traceability_index、YAML 预览和导出结果失效。
- 仅修改章节标题且正文和顺序不变：段落编号可以保留，但章节摘要、证据索引、Story Bible、Scene Plan 和后续产物需要标记为需复核。
- 段落编号变化：证据索引、Scene Plan、剧本 JSON、traceability_index、YAML 预览和导出结果失效。
- 风格来源在 Scene Plan 确认前被清空或重新选择：Style Profile、Scene Plan、剧本 JSON、YAML 预览和导出结果失效；需要重新生成 Style Profile 和 Scene Plan。
- Story Bible 变化：Scene Plan、剧本 JSON、traceability_index、YAML 预览和导出结果失效。
- Scene Plan 重新生成或重新确认：已生成剧本 JSON、traceability_index、YAML 预览和导出结果失效。
- 用户对话修改剧本：生成新的剧本版本、traceability_index、YAML 预览和导出结果；旧版本保留为历史版本，不作为当前版本。

失效产物不得删除原始记录。系统只改变其状态，例如 `current`、`stale`、`historical`、`failed`。

### 20.5 断点恢复

用户重新打开 session 后，可以：

- 查看上次进度。
- 从最近 checkpoint 继续。
- 查看历史版本差异。

第一版不支持版本回退，不提供回退到指定 checkpoint 的能力。历史版本和 checkpoint 用于断点恢复、差异查看和问题排查，不作为用户可操作的回滚功能。

## 21. 输出需求

### 21.1 内部输出

内部核心输出是 JSON。

JSON 用于：

- 前端展示定位。
- Schema 校验。
- 版本 diff。
- 断点恢复。
- 转换为其他格式。

内部剧本 JSON 必须包含 `content_block_id`，用于页面追溯来源证据。`content_block_id` 不进入用户最终导出文件。

### 21.2 必选输出

YAML 必须输出。

右侧成果区默认展示 YAML。该 YAML 由系统把内部 JSON 解析转换得到，仅用于只读预览和导出确认，不作为业务权威数据源。业务权威数据源仍是 PostgreSQL 中的内部 JSON 和相关结构化产物。

YAML 预览不可编辑。用户如果需要修改剧本，只能在对话中提出修改意见，由系统生成新的内部 JSON 版本后再重新渲染 YAML 预览。

YAML 必须符合预定义 Schema，并包含：

- 剧本标题。
- 人物表。
- 场景列表。
- 场景信息。
- 动作。
- 对白。
- 旁白。

右侧页面可以围绕 YAML 内容块展示来源证据标记或证据列表，用于告诉用户该内容块对应哪些证据；这些标记属于页面 UI 层，不属于 YAML 正文字段。

用户最终导出的 YAML 默认不包含来源证据编号、段落编号或内部追溯 ID。

系统内部必须保留 traceability_index，用于把 `content_block_id`、剧本场景、内容块和原文证据建立映射关系。页面中用户点击来源证据时，系统通过 traceability_index 弹出对应章节、段落和原文摘录。

### 21.3 可选输出

用户可选导出：

- Markdown。
- DOCX。
- DOC。
- PDF。
- TXT。
- 用户干净 JSON。

DOC 和 PDF 由 DOCX 导出内容通过 LibreOffice 转换得到。

导出层只转换用户可见的剧本内容，不改变剧本正文，不携带内部来源证据编号、段落编号或 traceability_index。

用户干净 JSON 与用户导出 YAML 同构，字段语义一致，只是格式不同。二者都不包含 `content_block_id`、来源证据编号、段落编号或 traceability_index。

导出时用户选择目标格式。可选格式不在主界面作为并列预览 tab 展示。

## 22. Schema 文档

系统必须输出 Schema 说明文档。第一版至少包含三套 Schema：

1. 内部剧本 JSON Schema。
2. 用户导出 YAML / 用户干净 JSON Schema。
3. traceability_index Schema。

用户导出 YAML 与用户干净 JSON 使用同一套用户可见内容 Schema，仅序列化格式不同。

文档应包含：

- Schema 设计目标。
- 内部剧本 JSON 顶层结构说明。
- 用户导出 YAML / 用户干净 JSON 顶层结构说明。
- traceability_index 顶层结构说明。
- 人物表字段说明。
- 场景列表字段说明。
- 内容块字段说明。
- `content_block_id` 的内部用途。
- 内部追溯索引说明。
- 为什么按场景组织。
- 为什么最终导出文件不包含来源证据编号。
- 页面如何通过 traceability_index 追溯原文证据。
- 校验规则说明。
- 可扩展字段说明。

Schema 边界：

- 内部剧本 JSON Schema 必须允许 `content_block_id`、`source_evidence_ids`、段落编号和内部追溯字段。
- 用户导出 YAML / 用户干净 JSON Schema 不允许出现 `content_block_id`、来源证据编号、段落编号或 traceability_index。
- traceability_index Schema 必须校验 `content_block_id`、scene_id、source_evidence_id、chapter_id、paragraph_id 之间的映射关系。

## 23. 前端页面

第一版采用单工作台界面，整体为左中右三栏结构，不以多页面跳转为主要交互方式。

三栏职责：

- 左侧：项目导航、当前项目主对话入口、Scene Plan 场景列表、生成剧本入口。
- 中间：Codex 风格对话区、风格来源选择、文件上传、Agent 执行进度。
- 右侧：当前项目成果查看区，根据左侧选择展示 Scene Plan 或生成后的 YAML 剧本。

### 23.1 左侧项目导航

应包含：

- `新建项目` 按钮。
- `项目` 列表。
- 当前项目名称。
- 当前项目 primary conversation 入口。
- 当前项目下的 `Scene Plan` 入口。
- `Scene Plan` 下方的场景列表。
- 当前项目下的 `生成剧本` 入口。

左侧项目可以折叠。用户点击项目时，系统展示该项目的当前对话和产物状态。

`Scene Plan` 是项目下的产物，不是项目本身。`Scene Plan` 也可以折叠，折叠后只显示入口名称，展开后显示已划分出的场景列表。

左侧不得混入其他项目的 Scene Plan 或剧本内容。当前项目为“雨夜归来”时，左侧只显示“雨夜归来”项目下的对话、Scene Plan 和生成剧本。

### 23.2 中间对话区

中间栏采用类似 Codex 的对话式体验。

应包含：

- 顶部横向风格来源小卡片。
- 用户与 Agent 的对话记录。
- Agent 执行进度块。
- 对话输入框。
- 文件上传入口。
- 当前可执行操作按钮。

上传小说文件放在对话输入框内，交互方式类似 Codex 上传附件。左侧不放上传小说入口。

风格来源放在对话框上方，以横向小卡片展示。用户在发送前选择一种风格来源：

- 内置风格卡片，例如现实主义、悬疑/惊悚、爱情/情感、喜剧、短剧/网剧。
- 自定义风格卡片，包含文字输入框和历史剧本文件上传入口。

风格来源第一版三选一，不同时启用：

- 选择一个内置风格。
- 在自定义风格卡片中输入文字描述。
- 在自定义风格卡片中上传历史剧本文件。

自定义风格卡片内的文字输入和历史剧本上传互斥：用户输入文字后，文件上传入口置为不可用；用户上传历史剧本后，文字输入框置为不可用。同一轮请求不允许同时提交文字风格描述和历史剧本文件。

当用户尚未上传小说或尚未选择风格来源时，中间对话区域用淡灰色提示文案显示：

```text
请上传小说并选择风格来源
```

用户上传小说并选择风格后发送请求，Agent 在对话区中显示执行进度。执行进度块应类似 Codex 的“正在思考/执行”状态，可以折叠和展开。

进度块至少展示：

- 正在解析上传文件。
- 正在识别章节。
- 正在生成章节摘要。
- 正在生成原文证据索引。
- 正在生成 Story Bible。
- 正在生成 Style Profile。
- 正在生成 Scene Plan。
- 正在等待用户确认 Scene Plan。
- 正在逐场生成剧本。
- 正在运行校验与修复。

### 23.3 右侧成果区

应包含：

- 用户点击左侧 `Scene Plan` 时：右侧展示当前项目的 Scene Plan。
- 用户点击左侧 `生成剧本` 时：右侧展示当前项目生成出的剧本。
- Scene Plan 确认前：右侧可以展示 Scene Plan 草稿。
- Scene Plan 确认后：右侧可以展示已确认 Scene Plan。
- 剧本生成后：右侧展示 YAML 只读预览。
- 页面默认不展示 `[剧本文本] [YAML] [JSON]` 并列预览 tab。
- 最终界面展示以 YAML 为主。
- 不同导出格式只在导出动作中选择。
- 来源证据入口属于页面 UI 层，不属于 YAML 正文内容。
- YAML 正文只展示剧本内容，不把来源证据按钮写入 YAML 字段。
- 页面可在场景标题、内容块旁或 YAML 预览外侧展示来源证据按钮、证据标记或证据列表，用来说明该内容块对应哪些来源证据。
- YAML 预览不可编辑。用户只能通过中间对话区提出修改意见。
- 用户点击想查看的来源证据后，弹窗展示对应章节、段落编号和原文内容。
- 来源证据弹窗右上角必须提供关闭按钮。
- 用户最终导出的文件不包含来源证据编号。

右侧只展示当前项目的成果，不展示其他项目内容。当前项目为“雨夜归来”时，右侧不能出现“旧城来信”的 Scene Plan 或剧本内容。

剧本尚未生成时，右侧显示空状态：

- 一个轻量、可爱的动态图标。
- 一段与当前流程同步的状态文案。

状态文案示例：

```text
正在等待用户上传
正在等待生成场景
正在等待用户确认 Scene Plan
正在等待生成剧本
```

右侧状态文案应与中间 Agent 执行进度保持一致。

如果某个 run 达到预算上限、自动修复失败或被标记为内部人工复核状态，普通用户前端不展示人工复核入口，但必须展示清晰失败状态，例如：

```text
本次生成未完成，请调整要求后重新发起
```

失败状态至少应允许用户：

- 查看失败发生的阶段。
- 保留当前已成功生成的内容。
- 通过对话调整要求后重新发起。

## 24. 验收标准

第一版通过验收需要满足：

1. 能新建项目。
2. 能上传至少 3 个章节的小说文本。
3. 能将输入文件统一转换为 Markdown。
4. 能识别并确认章节。
5. 能生成稳定段落编号。
6. 能生成章节结构化摘要。
7. 能生成原文证据索引。
8. 能生成 Story Bible。
9. 能选择内置风格、上传风格参考剧本或输入文字描述目标风格，并生成 Style Profile。
10. 能生成 Scene Plan，并支持用户确认。
11. 能逐场生成剧本 JSON。
12. 能导出合法 YAML。
13. 能输出内部剧本 JSON、用户导出 YAML / 用户干净 JSON、traceability_index 三套 Schema 说明文档。
14. 能进行 Validation Agent 校验。
15. 能进行程序化 Schema 校验。
16. 能为开发者保存 Prompt、工具调用、工具返回和校验报告，并提供开发人员查看方式。
17. 能保存 session 和 checkpoint。
18. 能从 checkpoint 恢复。
19. 能通过对话修改至少一个场景，并保存新版本。
20. 能在中间对话输入框上传小说文件。
21. 能在中间对话区上方横向展示风格来源卡片，其中自定义风格卡片同时包含文字输入和历史剧本上传入口。
22. 能折叠和展开左侧项目。
23. 能折叠和展开左侧 Scene Plan。
24. 点击左侧 Scene Plan 时，右侧能切换展示当前项目 Scene Plan。
25. 点击左侧生成剧本时，右侧能切换展示 YAML 只读预览。
26. 剧本未生成时，右侧能显示与流程进度一致的空状态文案。
27. 点击页面 UI 层来源证据入口时，能弹窗展示对应章节、段落编号和原文内容。
28. 普通用户前端不显示开发者模式、run 日志、完整 Prompt、工具参数或工具返回明细。
29. 用户确认 Scene Plan 后，前端风格来源保持锁定，不允许再次切换。
30. 前端不提供编辑 Style Profile 或风格规则的入口。
31. 用户导出文件不包含 `content_block_id`、来源证据编号、段落编号或 `traceability_index`。
32. 自定义风格卡片中文本输入和历史剧本上传互斥，不能在同一轮请求中同时提交。
33. YAML 预览不可编辑，用户修改必须通过对话提出。
34. 第一版不提供手动编辑 Scene Plan 字段的入口。
35. 可选导出的 JSON 必须是用户干净 JSON，不包含内部追溯字段。
36. 用户干净 JSON 与用户导出 YAML 同构，仅序列化格式不同。
37. 用户触发一次完整任务时创建一个 run_id，自动校验、修复和导出转换作为 run_step 记录。
38. 能记录 project/session、单 run、单 scene、LLM 调用、工具调用五类预算消耗。
39. 上游产物变化时，下游产物能标记为 stale，不继续作为当前有效产物参与生成。
40. 用户必须通过按钮或明确对话确认 Scene Plan，系统不得自动默认确认。
41. 失败或预算耗尽时，前端能展示失败阶段和重新发起提示，但不展示人工复核入口。
42. DOC 输入和 DOC/PDF 导出纳入验收，转换运行时由 Docker 镜像内置 LibreOffice 提供。

## 25. 风险与约束

### 25.1 长文本风险

风险：章节过长会导致上下文成本高、摘要遗漏、生成偏离。

应对：分块摘要、证据索引、逐场生成、覆盖率检查。

### 25.2 PDF 解析风险

风险：PDF 排版复杂或扫描版无法准确提取。

应对：第一版只支持文本型 PDF，推荐用户上传 TXT、DOC、DOCX 或 Markdown；扫描版 PDF 需先做 OCR。

### 25.3 风格误判风险

风险：Style Profile 可能过度概括用户风格。

应对：展示 Style Profile 提取结果，但不允许用户直接编辑风格规则。

### 25.4 校验误判风险

风险：Validation Agent 也可能遗漏问题。

应对：结合程序化 Schema Validator，并保留内部人工复核状态；第一版不做前端人工复核入口。

### 25.5 自动修复循环风险

风险：模型反复修复失败。

应对：严格限制修复次数，失败后标记内部人工复核状态；第一版不做前端人工复核入口。

## 26. 第一版成功标准

第一版成功不要求生成专业成品剧本，而要求证明系统具备：

- 长文本结构化处理能力。
- 原文证据追溯能力。
- 剧本 YAML 生成能力。
- 校验与修复能力。
- 对话式修改能力。
- 风格参考能力。
- session 保存与断点恢复能力。
- 开发者运行日志追踪能力。

一句话总结：

NovelScript AI 是一个由主控 Agent 编排多 Worker，并由独立 Validation Agent 校验的小说剧本改编工作台。
