## Frontend / Backend API Contract

This contract is the source of truth for frontend/backend collaboration during MVP implementation. Backend endpoints must return these shapes. Frontend code must consume these DTOs through `frontend/src/api/client.ts`.

### API Base Rules

Base URL:

```text
VITE_API_BASE_URL=http://localhost:8000
```

Transport:

```text
JSON requests and responses use application/json.
File uploads use multipart/form-data.
No authentication, account system, or multi-user permission model is implemented in MVP.
All IDs are opaque strings generated by the backend.
Timestamps are ISO 8601 strings.
Frontend must not call developer run log endpoints because no such frontend endpoint exists in MVP.
```

Common error response:

```json
{
  "error": {
    "code": "validation_error",
    "message": "Human-readable error message",
    "details": {}
  }
}
```

Common status values:

```ts
export type ArtifactStatus = "current" | "stale" | "historical" | "failed";
export type RunStatus = "queued" | "running" | "succeeded" | "failed" | "budget_exceeded";
export type ProjectStage =
  | "empty"
  | "uploaded"
  | "chapters_pending"
  | "chapters_confirmed"
  | "style_selected"
  | "scene_plan_draft"
  | "scene_plan_confirmed"
  | "script_generating"
  | "script_ready"
  | "failed";
```

#### Upload And Chapter APIs

```text
POST /projects/{project_id}/uploads
GET /projects/{project_id}/chapters/pending
POST /projects/{project_id}/chapters/confirm
```

`POST /projects/{project_id}/uploads` request:

```text
multipart/form-data

Single-file upload:
field: file

Multi-file upload:
field: files

Supported MVP extensions: .md, .txt, .docx, .pdf
.doc returns 415 unsupported_media_type
```

Upload splitting behavior:

- `file` supports single-document multi-chapter novels.
- `files` supports multi-document uploads. Files are sorted by natural filename order before chapter detection, for example `chapter2.txt` comes before `chapter10.txt`.
- Each uploaded document is converted to Markdown before chapter detection.
- A document with no recognizable chapter heading is treated as one chapter and uses the filename stem as the draft chapter title.
- All detected chapters are re-numbered into one project-level sequence: `CH001`, `CH002`, `CH003`.
- Paragraph IDs are generated per chapter after detection: `CH001_P001`, `CH001_P002`.
- Response shape is the same for single-file and multi-file uploads.

`POST /projects/{project_id}/uploads` response:

```json
{
  "file_id": "file_001",
  "project_id": "proj_001",
  "stage": "chapters_pending",
  "detected_chapters": [
    { "chapter_id": "CH001", "title": "第一章 雨夜", "order": 1, "paragraph_count": 12 }
  ]
}
```

`GET /projects/{project_id}/chapters/pending` response:

```json
{
  "chapters": [
    { "chapter_id": "CH001", "title": "第一章 雨夜", "order": 1, "paragraph_count": 12 }
  ]
}
```

`POST /projects/{project_id}/chapters/confirm` request:

```json
{
  "chapter_ids": ["CH001", "CH002"]
}
```

`POST /projects/{project_id}/chapters/confirm` response:

```json
{
  "project_id": "proj_001",
  "stage": "chapters_confirmed",
  "checkpoint_id": "chk_001"
}
```

#### Style Reference Upload APIs

```text
POST /projects/{project_id}/style-reference-uploads
```

`POST /projects/{project_id}/style-reference-uploads` request:

```text
multipart/form-data field: file
Supported MVP extensions: .md, .txt, .docx, .pdf
.doc returns 415 unsupported_media_type
```

Response:

```json
{
  "file_id": "file_style_001",
  "project_id": "proj_001",
  "purpose": "style_reference",
  "filename": "past-script.docx",
  "stage": "chapters_confirmed"
}
```

Rules:

```text
This endpoint stores a style-reference file only.
It does not create or replace the novel source file.
It does not change chapter detection, chapter confirmation, or paragraph IDs.
Uploading after Scene Plan confirmation returns 409 style_source_locked.
```

#### Style Source APIs

```text
POST /projects/{project_id}/style-source
GET /projects/{project_id}/style-source
DELETE /projects/{project_id}/style-source
```

`POST /projects/{project_id}/style-source` request examples:

```json
{ "kind": "builtin", "builtin_style": "suspense" }
```

```json
{ "kind": "custom_text", "style_text": "更悬疑，对白短促，节奏快。" }
```

```json
{ "kind": "reference_scripts", "reference_file_ids": ["file_style_001"] }
```

Rules:

```text
Only one style source kind is accepted per request.
reference_file_ids length must be between 1 and 3.
DELETE is allowed only before Scene Plan confirmation.
POST after Scene Plan confirmation returns 409 style_source_locked.
```

`POST /projects/{project_id}/style-source` response:

```json
{
  "project_id": "proj_001",
  "style_source": { "kind": "builtin", "builtin_style": "suspense" },
  "style_locked": false,
  "stage": "style_selected"
}
```

`GET /projects/{project_id}/style-source` response:

```json
{
  "project_id": "proj_001",
  "style_source": { "kind": "builtin", "builtin_style": "suspense" },
  "style_locked": false
}
```

If no style source has been selected:

```json
{
  "project_id": "proj_001",
  "style_source": null,
  "style_locked": false
}
```

`DELETE /projects/{project_id}/style-source` response: `204 No Content`.


```text

Frontend calls POST /projects/{project_id}/scripts/generate after Scene Plan confirmation.
```

```ts
export async function getActiveRun(projectId: string): Promise<AgentProgress | null>;
```

Frontend collaboration rules:

```text
All frontend components call backend only through api/client.ts.
Frontend stores no business authority copy of Scene Plan, script JSON, or traceability_index.
Frontend displays YAML as read-only text from getYamlPreview().
Frontend uses getCurrentScriptForUi() only to place evidence buttons and locate content_block_id markers.
Frontend evidence buttons pass content_block_id to getEvidenceByContentBlock().
Frontend must handle 409 scene_plan_not_confirmed, style_source_locked, and scene_plan_change_required.
```
